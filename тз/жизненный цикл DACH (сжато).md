### Техническое задание для Fable 5: B2B Micro-SaaS "Widerrufsbutton DACH"

**Контекст продукта:** Необходимо разработать универсальный встраиваемый JS-виджет и бэкенд-инфраструктуру для обеспечения соответствия интернет-магазинов параграфу § 356a BGB (Гражданский кодекс Германии), который вступает в силу 19 июня 2026 года.

**Технологический стек**

- **Frontend (Виджет):** Vanilla JavaScript (размер строго 3-5 KB). Обязательная инкапсуляция через Shadow DOM (режим `closed`).
    
- **Backend / БД:** Next.js API Routes (Vercel Edge Functions) + Supabase (PostgreSQL с RLS).
    
- **Криптография:** Модуль `crypto` Node.js, библиотека `@openzeppelin/merkle-tree` и интеграция с DigiCert TSA по протоколу RFC 3161.
    
- **Email:** Resend API или SendGrid со строгой настройкой DKIM/SPF.
    
- **Хостинг / CDN:** Vercel для API и Cloudflare/AWS CloudFront для отдачи минифицированного скрипта `widget.min.js`.
    

**Жизненный цикл проекта**

**1. Инициация (Идея и планирование)**

- **Цель:** Заложить юридический и архитектурный фундамент продукта.
    
- **Основные шаги:**
    
    - Проектирование схемы полезной нагрузки (JSON Payload schema) с соблюдением минимизации данных GDPR.
        
    - Разработка логики работы без авторизации пользователя (Guest Checkout Bypass).
        
    - Подготовка юридических текстов для UI на немецком и английском языках.
        
- **Результат:** Утвержденная архитектура данных и полная схема пользовательского пути (User Journey).
    

**2. Проектирование (Планирование архитектуры)**

- **Цель:** Спроектировать отказоустойчивую систему, интегрируемую через единый тег `<script>`.
    
- **Основные шаги:**
    
    - Проектирование изоляции стилей: агрессивные CSS-сбросы сайтов не должны ломать виджет, а виджет не должен влиять на глобальные стили сайта.
        
    - Проектирование БД в Supabase со строгими таблицами: `organizations`, `withdrawal_requests`, `merkle_roots`.
        
    - Разработка спецификации OpenAPI/Swagger с учетом CORS и Rate Limiting.
        
- **Результат:** Готовые миграционные SQL-скрипты и задокументированный протокол интеграции.
    

**3. Реализация (Разработка) — КРИТИЧЕСКАЯ ФАЗА**

- **Цель:** Написание production-ready кода фронтенда, бэкенда и фоновых воркеров.
    
- **Основные шаги:**
    
    - **Фронтенд:** Разработка Custom Web Component с использованием `attachShadow({ mode: 'closed' })` и двухступенчатым модальным окном.
        
    - **Бэкенд:** Написание защищенного Edge API для приема заявок с валидацией через Zod и санитизацией.
        
    - **Криптография:** Генерация канонического JSON и вычисление SHA-256 хеша до записи в БД. Написание фонового CRON-микросервиса (запуск в 23:59 UTC) для объединения хешей в Merkle Tree и получения токена от DigiCert TSA.
        
- **Результат:** Стабильный прототип с настроенными политиками RLS в Supabase и автоматическим криптографическим хешированием.
    

**4. Внедрение (Деплой и запуск)**

- **Цель:** Вывод в Production с обеспечением высокой доступности (High Availability).
    
- **Основные шаги:**
    
    - Деплой бэкенда как Vercel Edge Function (с гео-распределением, например, во Франкфурте `fra1`).
        
    - Сборка скрипта через Webpack/Rollup и настройка агрессивного кэширования на CDN.
        
    - Создание Client Dashboard для настройки доменов (CORS whitelist) и получения API-ключа.
        
- **Результат:** Масштабируемый SaaS, выдерживающий пиковые нагрузки без потери транзакций.
    

**5. Поддержка и развитие**

- **Цель:** Масштабирование выручки (MRR) и поддержание комплаенса.
    
- **Основные шаги:**
    
    - Внедрение портала для экспорта судебных доказательств (ZIP-архив с JSON-логом, Merkle Proof и RFC 3161 токеном).
        
    - Разработка White-label (B2B2B) функционала для IT-агентств и интеграторов.
        
    - Внедрение Dual-Button Widget — алгоритмическое разделение UI для Kündigungsbutton (§ 312k BGB) и Widerrufsbutton (§ 356a BGB).
        
- **Результат:** Самоподдерживающийся B2B-актив с минимальным Churn Rate.
    

**Юридическая сторона и Архитектура (Внимание для Fable 5)**

Fable 5, при написании кода ты обязан руководствоваться принципом «Zero Trust Environment» и строго соблюдать следующие требования:

**1. Строгий 4-этапный UI-процесс:**

- **Этап 1 (Entry Button):** Кнопка вызова должна содержать текст "Vertrag widerrufen". Доступ категорически запрещено блокировать Login-Wall.
    
- **Этап 2 (Форма):** Сбор данных. Запрашивать ТОЛЬКО имя, номер заказа/контракта и email. АБСОЛЮТНО ЗАПРЕЩЕНО добавлять поле "причина возврата".
    
- **Этап 3 (Confirmation Button):** Финальная кнопка отправки должна содержать текст "Widerruf bestätigen".
    
- **Этап 4 (Instant Receipt):** Мгновенная автоматическая отправка email-квитанции потребителю с указанием точного времени получения сервером.
    

**2. Бремя доказывания (Beweislast) и Аудит:**

- Обычные SQL-логи не имеют юридической силы. Требуется Tamper-Evident Audit Log.
    
- Бэкенд должен генерировать SHA-256 хеш на основе канонического JSON. Хеш обязан включать: ID магазина, точный номер заказа, email и неизменяемый Timestamp в формате ISO 8601.
    
- Клиенту в email отправляется публичная ссылка (Proof-URL) на этот лог.
    
- Обязательна реализация паттерна "exponential backoff" при сбоях связи с внешним TSA-сервером. В случае полного сбоя, лог сохраняется с флагом `tsa_pending: true` для отложенной нотаризации.
    

**3. Изоляция (Shadow DOM):**

- Интерфейс обязан рендериться внутри изолированного дерева, чтобы внешние CSS-сбросы клиента не сделали кнопку расторжения нечитаемой.
